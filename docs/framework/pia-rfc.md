# PIA RFC：面向用户的 PI 角色

> [!abstract]
> `PIA`（principal investigator agent）是工作在 orchestrator 外层、面向用户的 PI 角色。它不直接替代容器内研究执行，而是读取系统状态、日志与阶段产物，将研究进展转换为用户可理解的解释、预算与范围沟通、阶段汇报和管理性文本。

## 定位

- `PIA` 负责对接用户，而不是直接做容器内研究。
- 它的工作重点是解释、沟通、重述和包装，而不是拆研究任务到执行原子步骤。
- 在角色图景上，它更接近“院士大导师”：负责与外界沟通需求、解释当前进展、确定预算与优先级，并产出项目申请或基金申请等管理性文本。

## 非目标

- 不直接组织一个或多个 `VSA` 的日常研究推进，这属于 `supervisor`。
- 不直接承担论文细读、轻量实验或容器内工作区记账，这属于 `VSA`。
- 本轮不展开首批 tools / skills 清单，只固定职责、能力域和典型产物。

## 核心职责

- 与用户澄清需求，重述目标、边界与期望结果。
- 读取 orchestrator、`supervisor` 和 `VSA` 的状态、日志、工件与阶段成果。
- 向用户解释当前任务的进展、已有成果、剩余问题和下一步方向。
- 沟通预算、范围、优先级和人工介入时机。
- 为项目申请、基金申请、阶段汇报等管理性文本提供起草支持。

## 能力域

### 1. 需求澄清与目标重述

- 把用户的问题转译为研究目标、阶段目标和可沟通的任务边界。

### 2. 进展解释与状态转译

- 基于 logs、artifacts、阶段状态和 supervisor 汇报，回答“现在在做什么”“做到了哪里”“谁在做什么”。

### 3. 预算、范围与优先级沟通

- 向用户解释为什么需要扩大/收缩预算、为什么要延后某条路线、当前最值得推进什么。

### 4. 阶段成果汇报与对外叙事

- 把容器内研究资产整理成面向用户、管理者或合作方的阶段成果说明。

### 5. 管理性文本起草

- 起草基金申请书、项目申请材料、阶段总结、任务说明和其他需要对外提交的文本。

### 6. 人工介入建议整理

- 当 `supervisor` 认为研究阻塞、预算触顶或需要决策切换时，整理成面向用户的介入建议与决策选项。

## 典型产物

- 需求澄清摘要
- 当前进展说明
- 阶段成果汇报
- 预算与范围建议
- 人工介入建议
- 基金申请书或项目申请草稿

## 与其他角色的关系

- `PIA <- supervisor`
  - 接收研究推进摘要、风险提示、预算压力和阶段成果。
- `PIA -> 用户`
  - 解释系统当前做了什么、为什么这么做、还缺什么、需要用户决定什么。
- `PIA` 不直接替代 `supervisor` 的研究管理职责，也不直接替代 `VSA` 的容器内研究职责。

## 当前规划边界

- 本轮只把 `PIA` 定义到角色职责、能力域和典型产物层级。
- 后续若需要为 `PIA` 建能力基座，再单独规划其行为约定、对话协议、日志读取面和文案工具面。
