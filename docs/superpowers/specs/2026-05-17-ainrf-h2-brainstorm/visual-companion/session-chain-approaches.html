Session Chain — 三种实现路径

从轻到重，对 AINRF Task 模型的改造程度不同

A

轻量级：扩展现有 Task 字段

给 tasks 表加 session_id + continued_from_task_id

tasks 表新增:
· session_id TEXT — 同一次研究会话的标识
· continued_from_task_id TEXT — 指向前一次尝试

Pros

  • 改动最小，一个 migration 搞定
  • 复用现有 Task API / TaskList / TaskDetail
  • 向后兼容——旧 task 字段留空即可

Cons

  • Session 没有独立 API——聚合查询需在应用层做
  • Session 级别的元数据（总耗时、总 cost）需实时计算
B

中等：TaskAttempt + 独立 Session 视图

Task 表示"逻辑任务"，TaskAttempt 表示"每次执行"

task_sessions 新表:
· id, title, project_id, created_at, status

task_attempts 新表 (或复用 tasks 表):
· id, session_id, task_id (可选), attempt_seq
· parent_attempt_id, intervention_reason
· status, started_at, finished_at, duration_ms

Pros

  • Session 是一等公民，有独立 API 和 UI
  • 逻辑任务 vs 执行尝试清晰分离
  • Session 级别聚合（cost、耗时）可持久化

Cons

  • 新表 + 新 API + 新前端页面，工作量大
  • 需要迁移现有 Task 数据或提供兼容层
C

完整版：AgentSession（humanize2 对齐）

Session 为顶层实体，含 runs 列表、token 统计、中断链

agent_sessions:
· id, title, project_id, agent_type, model
· total_input_tokens, total_output_tokens
· total_cost_usd, attempt_count
· created_at, updated_at

agent_runs (替代 tasks 的部分职能):
· session_id, parent_run_id, continued_from_run_id
· intervention_message, backend_session_id
· output_events (JSONL), token_usage

Pros

  • 完全对齐 humanize2 的 Session 模型
  • Timeline / Token / Session 全部原生支持

Cons

  • 与现有 Task 模型大幅重叠，引入两种平行的"运行"概念
  • AIRNF 的 Task 承载了 workspace/env binding 等额外职责
  • 过度设计——AINRF 不需要 humanize2 的全部复杂度
