三个概念解释：Cartridge、Session Chain、Token 追踪

Humanize2 中的关键设计模式

1. HTML Cartridge 工作流引擎

用 HTML 自定义标签描述多 Agent 协作流程，由引擎解析成 DAG 并执行。

Cartridge 源文件（HTML）
<h2-workflow id="rlcr">
  <h2-flow>
    <h2-agent agent="claude">
      实现需求
    </h2-agent>
    <h2-agent agent="codex">
      审查代码
    </h2-agent>
    <h2-loop max="3">
      ...修复循环...
    </h2-loop>
  </h2-flow>
</h2-workflow>
编译后的执行 DAG
Start
↓
Claude: 实现
↓
Codex: 审查
↓
Loop (max 3)
Claude Fix → Codex Re-Review
↓
End

Cartridge 的优势

  • 流程定义即文档——HTML 可读可执行
  • 支持 loop / parallel / branch / await 控制流
  • Expression 引擎实现条件路由
  • Cartridge 可热加载、可版本管理

对 AINRF 的适用性

  • AINRF 已有 Task Edge DAG，Cartridge 是更高级的 DSL
  • 短期内可能 over-engineered
  • 但 Loop + Parallel + Condition 的能力值得参考

2. Session Chain（会话链）

每次 Agent Run 通过 continuedFromRunId 串联，形成完整的中断-继续链路。

Session Chain 示意
Session A
Run #1
实现功能
12m 30s
→
Session A
Run #2
修复审查意见
8m 15s
→
Session A
Run #3
最终通过 ✓
5m 42s
同一个 AgentSession（root parent 聚合）→ Timeline 上显示为一行的多个色块

对 AINRF 的价值

  • AINRF 的 Task 之间可以通过 Task Edge 连接，但没有原生的 "同一个任务的多次尝试" 概念
  • Session Chain = 一次研究任务被中断/继续/修正的完整历史
  • 可直接支撑 Timeline 的时间轴渲染

实现考量

  • 需要给 Task 增加 session_id 和 parent_task_id
  • 或引入更轻量的 TaskAttempt 概念

3. Token 用量追踪

Humanize2 在 Agent Run 的 output events 中提取 token 信息，按 Session 聚合展示。

Token Flow 展示（概念）
Session A · Claude Opus 4.7 Total: $3.42
Input 48K
Cache 22K
Output 8K
Think 3K
Session B · Codex GPT-5.5 Total: $1.87
Input 32K
Cache 14K
Output 12K
Think 5K
← 横条宽度 = token 占比，颜色区分 input / cache / output / reasoning

对 AINRF 的价值

  • AINRF 已有 TaskOutputKind（含 message / thinking / tool_call），token 数据可能已经存在
  • 研究者关心每个研究任务的 API 开销
  • 可按 Project / Workspace 聚合成本

实现考量

  • 需从 agent-sdk engine 的 event stream 中解析 token usage
  • Claude CLI 模式下 token 数据可能不可用
