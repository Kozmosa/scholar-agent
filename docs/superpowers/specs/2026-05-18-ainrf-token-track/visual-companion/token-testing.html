Token Track — Testing & Implementation Order

Section 5/5 · 测试范围 + 6 步实现顺序

Testing Scope

Backend (pytest)

  • EngineEvent.token_usage 字段序列化/反序列化
  • AgentSdkEngine 从 ResultMessage 提取 token_usage
  • ClaudeCodeEngine session-meta 轮询（含超时处理）
  • TaskOutputKind.TOKEN 事件写入 output_events
  • _run_task 中 token_usage 捕获 → complete_attempt 调用
  • _recalc_session cost 聚合正确性（含 NULL 处理）
  • SessionService 注入为 None 时 task harness 不受影响
  • GET /projects/{id}/cost-summary

Frontend (Vitest)

  • TokenFlowBar 组件渲染（4 色比例正确）
  • TokenFlowBar 空态（无 token 数据）
  • TokenFlowBar Claude Code 态（仅 input/output）
  • SessionDetail 展示 total_cost_usd + total_tokens
  • Per-Model Breakdown 展开/折叠
  • AttemptChain 集成 TokenFlowBar

Implementation Order

Step 1 EngineEvent 扩展 + TaskOutputKind.TOKEN + AgentSdkEngine token 提取
Step 2 Task Harness 集成: SessionService 注入 + emit 捕获 + complete_attempt 写入
Step 3 ClaudeCodeEngine session-meta 轮询提取
Step 4 _recalc_session cost 聚合 + Project cost-summary API
Step 5 前端: TokenFlowBar + AttemptChain 集成 + SessionDetail 升级
Step 6 前端: Per-Model Breakdown + Per-Project Cost 展示 + i18n

Key Risks & Mitigations

Risk Mitigation
Claude Code session-meta 匹配不准确 按 task 启动时间 ± 5 秒窗口匹配；匹配失败时 token_usage=None（不阻塞）
SDK usage 字段未来改名 用 .get() 安全访问；source 字段标记数据来源便于排查
SessionService 注入导致循环依赖 SessionService 不依赖 TaskHarnessService；单向引用，安全
